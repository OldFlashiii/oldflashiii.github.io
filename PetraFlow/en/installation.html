Installation#

This page focuses on the runtime environment needed to use PetraFlow: the Julia project environment, dependencies, optional solvers, and minimal sanity checks.

Requirements#

Julia 1.10 or newer is recommended. PetraFlow depends on Julia packages such as JuMP, MathOptInterface, and Ipopt. For LP/QP/SOCP/MISOCP or mixed-integer models, additional solver backends such as Gurobi or HiGHS can be configured.

The expected project layout is:

PetraFlow/
  Project.toml
  Manifest.toml
  src/
  data/
  test/

The commands below assume that you run Julia from the PetraFlow project root.

Activate the Julia Project#

In the Julia REPL:

using Pkg
Pkg.activate(".")
Pkg.instantiate()

Pkg.activate(".") makes Julia use the current project’s Project.toml; Pkg.instantiate() restores dependencies from Manifest.toml. The first run may take some time. After dependencies are ready, loading PetraFlow is simply:

using PetraFlow

If the first load is slow, Julia is usually precompiling dependencies. Later loads are normally faster.

Solver Backends#

PetraFlow optimization models call solvers through JuMP/MOI. Different formulations require different solver capabilities:

Problem class

Common solvers

Typical components or tasks

NLP

Ipopt

ACPPowerFlow(), ACRPowerFlow(), IVRPowerFlow(), continuous taps, nonlinear converters

LP/QP

Gurobi, HiGHS

DCOPF-QP, PTDF OPF, linear load shedding, selected linearized devices

SOCP

Gurobi

WSOCPowerFlow(), branch-flow SOCP, conic device limits

MILP/MISOCP

Gurobi

Unit commitment, switches, discrete shunts, selected storage exclusivity models

PetraFlow internal algorithms

:petraflow

NLP/QP interior-point methods, structured KKT systems, solver research

Check solver availability with:

optimizer_available(:ipopt)
optimizer_available(:gurobi)
solver_candidates(:NLP)
solver_candidates(:SOCP)

If a solver is unavailable, the modular interface can try to select an available backend from the problem class. For benchmark work, explicitly specify the solver and attributes.

Minimal Sanity Checks#

After installation, run two small checks. First, load a MATPOWER case and solve a power flow:

using PetraFlow

net = load_network("case14")
pf = solve_pf(net; method = :NR, verbose = false)

pf.success
pf.max_mismatch

Second, check that the modular modeling interface can create a context and initialize state variables:

power = load_power_network("case14")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)
has_variable(ctx, :power, :vm)
has_variable(ctx, :power, :va)

If both checks run, data loading, basic power flow, and modular initialization are ready.

Data Directory#

Example data is available under PETRAFLOW_DATA_DIR:

joinpath(PETRAFLOW_DATA_DIR, "PGLIB", "pglib_opf_case118_ieee.m")
joinpath(PETRAFLOW_DATA_DIR, "PGLIB-HVDC")

Custom MATPOWER or PGLIB files can be loaded by passing their paths:

net = load_network(raw"C:\path\to\case.m")
power = load_power_network(raw"C:\path\to\case.m")

Common Issues#

Symptom

Possible cause

Suggestion

First using PetraFlow is slow

Julia is precompiling dependencies

Wait for precompilation; later loads should be faster

optimizer_available(:gurobi) returns false

Gurobi.jl, license, or environment variables are not configured

Start with Ipopt/HiGHS, then check the Gurobi installation

ACOPF does not converge

Starts, limits, data, or solver attributes may be unsuitable

Run runpf first, inspect data, adjust tolerances, or try another formulation

AC/DC case has no DC variables

Default network type is :ac

Set network_type = :acdc in init_power!
