Modeling and Solving Overview#

This page explains how PetraFlow connects power-network data, steady-state analysis, OPF workflows, solver research, and modular modeling. It is written for users who want to understand which API to call first, and then how the underlying pieces are separated.

PetraFlow organizes both documentation and source code around user workflows: Manual explains how to use the package, Library collects APIs and model catalogs, and Developer explains how to extend components. The project emphasizes three lines:

  1. Power flow and steady-state analysis: analyze a network without building an optimization model.

  2. OPF and solver-platform algorithms: use runopf / solve_opf for standard workflows, or the internal NLP/QP APIs for algorithm research.

  3. Modular optimization modeling: build a JuMP model from state-space initialization, reusable components, solver routing, and semantic result queries.

Line 1: Power Flow and Steady-State Analysis#

The steady-state layer does not require a JuMP model. It works directly with a typed Network to compute voltages, branch flows, admittance matrices, Jacobians, sensitivities, and security-analysis outputs.

flowchart TB load["load_network / build_network"] mats["make_ybus / make_bdc / makeB"] pf["solve_pf / runpf"] sens["make_jac / dSbus_dV / make_ptdf / make_lodf"] security["run_cpf / static_security_analysis / run_n1_analysis"] result["voltage / generation / branch_flow / losses / security_summary"] load --> mats load --> pf --> result pf --> sens mats --> sens sens --> security --> result

Data and Networks#

Function

MATPOWER transformers and shunts are represented in the typed network:

  • Branch.tap and Branch.shift represent off-nominal tap ratios and phase shifters.

  • Bus.gs and Bus.bs represent fixed active/reactive shunts.

  • make_ybus accounts for series admittance, line charging, taps, phase shifts, and bus shunts.

Line 2: OPF and Solver-Platform Algorithms#

The OPF path has two levels. Regular users should start with runopf / solve_opf. Users studying KKT systems, QP fast paths, or interior-point methods can then move into NLPProblem, QPProblem, and src/solver.

flowchart TB runopf["runopf / solve_opf"] acopf["ACOPF\nACP / ACR / IVR"] dcopf["DCOPF-QP\nB-theta / PTDF"] external["External solvers\nIpopt / Gurobi / HiGHS"] internal["PetraFlow solvers\nPDIPM / QP IPM / KKT routing"] report["report_solution / collect_solver_metrics / logs"] runopf --> acopf runopf --> dcopf acopf --> external acopf --> internal dcopf --> external dcopf --> internal external --> report internal --> report

One-Line OPF#

result = runopf(

    formulation = :acp,

Line 3: Modular Optimization Modeling#

The modular layer lets users compose models from reusable pieces instead of writing all JuMP variables and constraints in notebooks.

flowchart TB network["load_power_network / build_power_network"] ctx["power_context\nJuMP.Model + networks + registry"] init["init_power!\nformulation + state_variables"] comp["add_modules!\nPowerFlow + Limits + Cost + Devices + Coupling"] build["build_modular_model!\nvariables -> constraints -> objective"] solve["solve_modular_model!\nsolver routing + optimize!"] inspect["solution_status\nsolution_value\nbus_solution / branch_solution"] network --> ctx --> init --> comp --> build --> solve --> inspect

User Workflow#

power = load_power_network("case14")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)
add_modules!(ctx, ACPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

solution = solve_modular_model!(
    ctx;
    solver = :ipopt,
    attributes = Dict("print_level" => 0),
)

Solver Routing and Result Queries#

The modular layer infers the whole problem class from component declarations:

infer_modular_problem_class(ctx)
select_optimizer(ctx)
solve_modular_model!(ctx; solver = :auto)

After solving, use semantic result queries:

These helpers hide matrix dimensions and p.u. conversions, which makes them better suited for notebooks and user scripts.

Which Route Should I Choose?#

Goal

Recommended route

Quickly solve a power flow

runpf("case14")

Inspect Ybus, PTDF, or LODF

load_network + make_ybus / make_ptdf / make_lodf

Run a standard ACOPF benchmark

runopf(...; formulation = :acp)

Compare ACP/ACR/IVR

power_context + init_power! + the matching *PowerFlow() component

Run DCOPF-QP algorithm experiments

build_dcopf_qp + solve_dcopf_qp or QPProblem

Add storage, HVDC, FACTS, or UC

Modular modeling plus the corresponding components

Add a new device formulation

Create an AbstractOptimizationComponent subtype and implement the component protocol
