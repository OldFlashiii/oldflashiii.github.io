PetraFlow Architecture#

PetraFlow follows the same high-level modeling principle that makes PowerModels.jl useful for research: separate the problem specification from the network formulation, then attach reusable modeling components for devices, limits, costs, security constraints, and cross-domain coupling.

Source Layout#

The modular modeling source lives under src/power_system/optimization/. core/ contains the component protocol, build context, registry, solver routing, and result recovery. networks/ contains PowerNetwork, CyberNetwork, GasNetwork, HeatNetwork, and network bundles. modules/ contains the components that actually register JuMP variables, constraints, and objective terms.

modules/ is organized by modeling object:

Directory

Boundary

power_flow/

Network formulations: ACP, ACR, IVR, DCP, PTDF, W-SOC, W-SDP, QC, branch-flow, LinDistFlow

power_devices/

Physical power-system devices and limits: generators, branches, loads, storage, shunts, transformers, FACTS, HVDC, DC grid, switches, EV

objectives/

Cost and penalty terms: generation cost, PWL cost, load shedding, storage degradation, branch overload, multi-energy cost

security/

Contingency and corrective redispatch components

cyber/

Communication, measurement, and cyber-physical coupling

integrated_energy/

Gas, heat, and multi-energy coupling components

power_devices/basic_power_devices.jl keeps the common core components. modules/deterministic_power_components.jl is the module-layer deterministic component include entry and only controls include order. Optional device formulations live in grouped files under power_devices/deterministic/. power_devices/deterministic_power_components.jl is kept only as a compatibility shim for old include paths. New components should be added according to the physical object they model so the module tree remains readable.

Three User Routes#

Route A: MATPOWER-like direct analysis#

Use high-level calls when you want a fast power-flow or OPF run:

pf = runpf("case118"; method = :NR, verbose = false)
opf = runopf("case118"; formulation = :acp, solver = :ipopt)

Route B: modular optimization modeling#

Use the modular interface when you need to compare formulations or combine storage, HVDC, FACTS, unit commitment, or security components:

power = load_power_network("case118")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)
add_modules!(ctx, ACPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

result = solve_modular_model!(ctx; solver = :ipopt)

Route C: structured solver research#

Use the solver core when you study KKT routing, interior-point methods, QP presolve, scaling, restoration, or acceleration strategies. This layer can consume OPF instances, modular JuMP/MOI models, or standalone NLP/QP problems.

Component Protocol#

Every modular component implements the same protocol:

Function

Role

component_name(component)

Registry and report name

formulation_short_name(component)

Mathematical formulation name, such as PF-AC-Polar

formulation_problem_class(component)

LP, QP, NLP, SOCP, MILP, MISOCP, MINLP, etc.

required_networks(component)

Required networks, such as :power, :cyber, :gas, :heat

required_variables(component)

Variables that must already exist

provided_variables(component)

Variables introduced by the component

register_variables!

Write variables into the JuMP.Model

register_constraints!

Write constraints

register_objective!

Write objective terms

recover_result

Recover component-level results

:voltage_square

v, l, P, Q

Branch-flow SOC relaxation

Hybrid AC/DC networks must be requested explicitly:

init_power!(
    ctx;
    formulation = :acp,
    state_variables = :polar,
    network_type = :acdc,
    dc_state_variables = :voltage_square,
)

The default is network_type = :ac, so pure AC cases do not create DC-side variables.
