Modular Optimization#

The modular layer builds one JuMP model from independent components. Users choose the network state space first, then add physical and objective components.

Workflow#

Step

Function

Purpose

1

load_power_network or build_power_network

Create a power network

2

power_context or modular_context

Create an OptimizationBuildContext

3

init_power!

Initialize state variables

4

add_modules!

Add power-flow, device, cost, security, or coupling components

5

solve_modular_model!

Select/configure a solver and optimize

 
power = load_power_network("case14")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :dcp, state_variables = :dc)
add_modules!(ctx, DCPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

result = solve_modular_model!(ctx; solver = :gurobi, attributes = Dict("OutputFlag" => 0))

Context Contents#

OptimizationBuildContext stores the JuMP model, network bundle, registered variables, registered constraints, objective terms, components, and metadata.

Useful helpers:

has_variable(ctx, :power, :Pg)
get_variable(ctx, :power, :Pg)
component_summaries(ctx)
infer_modular_problem_class(ctx)

After optimization, prefer the solution query helpers over direct JuMP.value.(get_variable(...)) calls:

solution = solve_modular_model!(ctx; solver = :ipopt)

solution_status(solution)
modular_solution_summary(solution)
available_solution_variables(solution)

solution_value(solution, :Pg; index = 1, time = 1)
variable_solution(solution, :Pbr_from; time = 1)

bus_solution(solution; time = 1)
generator_solution(solution; time = 1)
branch_solution(solution; time = 1)
storage_solution(solution; time = 1)
power_solution(solution; time = 1)

solution_value returns registered variable values in their native modeling units, typically p.u. for power-system variables. The semantic table helpers return rows organized by buses, generators, branches, and storage devices, with active/reactive/apparent power converted to MW, MVAr, and MVA.

Query

Typical fields

bus_solution

vm_pu, va_deg, p_demand_mw, p_generation_mw, load_shed_mw

generator_solution

pg_mw, qg_mvar, on, start, stop, pmin_mw, pmax_mw

branch_solution

p_from_mw, q_from_mvar, p_to_mw, s_from_mva, loading_percent, theta_diff_deg

storage_solution

soc, energy_mwh, charge_mw, discharge_mw, net_power_mw

State Variables#

Formulation

State choices

Typical use

:acp

:polar, [:V, :theta]

Polar ACOPF

:acr

:rectangular, [:Vr, :Vi]

Rectangular ACOPF

:ivr

:ivr

Current-voltage ACOPF

:dcp

:dc, :angle

DCOPF

:ptdf

:auto

PTDF-based DCOPF

:wsoc

:w_space

W-space conic relaxation

:bf_soc

:voltage_square

Branch-flow conic relaxation

The initializer can also be used to create extra state variables explicitly:

init_power!(ctx; formulation = :acp, state_variables = [:V, :theta, :Vr, :Vi])

This is useful for experimental models that need more than one voltage coordinate system.

Component Registry#

registered_optimization_modules()
find_optimization_modules(:storage)
find_optimization_modules(:hvdc)

Each descriptor reports the component name, formulation, problem class, required variables, provided variables, and network dependencies.
