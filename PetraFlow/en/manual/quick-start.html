Quick Start#

This page walks through a complete PetraFlow workflow: load a network, solve a power flow, build a small interpretable network, assemble a modular ACOPF, switch formulations, and query results. The examples use user-facing APIs rather than internal data structures.

1. Load PetraFlow#

Start Julia from the project root, activate the environment, and load the package:

using Pkg
Pkg.activate(".")

using PetraFlow

If this is your first run, see Installation to check the Julia environment and solver backends.

2. Load a Standard Case#

PetraFlow can load built-in MATPOWER/PGLIB cases or user-provided .m files. The smallest example is IEEE 14-bus:

net = load_network("case14")

net is a typed Network; buses, generators, branches, storage devices, and costs are represented as Julia records. Inspect the case before solving:

nbus(net)
ngen(net)
nbranch(net)
total_load(net)
total_generation(net)

This step is not optimization. It checks that the input data has been parsed correctly. For new cases, it is useful to inspect network size, total demand, generator count, transformer count, and islanding.

3. Solve the First Power Flow#

Power flow is the basic steady-state analysis. Given injections and voltage set points, PetraFlow can solve voltages and branch flows with Newton-Raphson and other methods:

pf = solve_pf(net; method = :NR, verbose = false)

pf.success
pf.iterations
pf.max_mismatch

If pf.success is true, the power flow converged. Query results with semantic helpers:

voltage(pf)
generation(pf)
branch_flow(pf)
losses(pf)

For more steady-state analysis:

make_ybus(net)
make_ptdf(net)
make_lodf(net)
static_security_analysis(net)

4. Build a Small Network#

Standard cases are useful for benchmarks, but small interpretable systems are better for teaching, tests, and component validation. The builder API lets you add devices one by one:

power = build_power_network(name = "two_bus", base_mva = 100.0) do b
    add_bus!(b, 1; type = :ref, vm = 1.0)
    add_bus!(b, 2; type = :pq, pd = 80.0, qd = 20.0)

    add_generator!(
        b,
        1;
        pg = 80.0,
        qg = 20.0,
        pmax = 150.0,
        pmin = 0.0,
        qmax = 80.0,
        qmin = -80.0,
    )

    add_transformer!(b, 1, 2; r = 0.01, x = 0.05, tap = 1.05, shift = 3.0)
    add_capacitor!(b, 2; q_mvar = 5.0)
    add_reactor!(b, 2; q_mvar = 1.0)
end

add_transformer! creates a branch with tap ratio and phase shift. add_capacitor! and add_reactor! write bus shunts. You can still recover the typed Network:

net2 = power.data
make_ybus(net2)
count_transformers(net2)
total_shunt(net2)

Small systems are useful when you need to verify that one device parameter enters Ybus or the optimization model correctly.

5. Solve a Modular ACOPF#

Modular modeling starts from a PowerNetwork. The following example loads a PGLIB 118-bus case, initializes polar ACOPF variables, and adds power flow, generator limits, branch limits, and generation cost:

power = load_power_network(joinpath(PETRAFLOW_DATA_DIR, "PGLIB", "pglib_opf_case118_ieee.m"))
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)

add_modules!(
    ctx,
    ACPPowerFlow(),
    GeneratorLimits(),
    BranchLimits(),
    GenerationCost(),
)

These lines correspond to the core ACOPF structure:

Step

Purpose

power_context

Create the optimization context with a JuMP model, network, and registries

init_power!

Create base variables from the selected formulation and state space

ACPPowerFlow()

Add AC polar power-balance and branch-flow equations

GeneratorLimits()

Add active/reactive generator bounds

BranchLimits()

Add branch-capacity constraints

GenerationCost()

Add generation-cost objective terms

Solve the model:

result = solve_modular_model!(
    ctx;
    solver = :ipopt,
    attributes = Dict("print_level" => 0, "tol" => 1.0e-7),
)

6. Query Results#

Start with solver status:

solution_status(result)
result.termination_status
result.primal_status
result.selection

Then query power-system objects:

bus_solution(result; time = 1)
generator_solution(result; time = 1)
branch_solution(result; time = 1)

You can also query a specific registered variable:

available_solution_variables(result)
solution_value(result, :power, :vm; index = 1, time = 1)
solution_value(result, :power, :pg; index = 1, time = 1)

Semantic result queries are part of the modular modeling design. Users should be able to inspect bus, generator, branch, storage, and HVDC quantities without manually searching through raw JuMP containers.

7. Switch Formulations#

The same network and device components can be used with different state-variable spaces. Polar ACOPF:

ctx = power_context(power; T = 1)
init_power!(ctx; formulation = :acp, state_variables = :polar)
add_modules!(ctx, ACPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

Rectangular ACOPF:

ctx = power_context(power; T = 1)
init_power!(ctx; formulation = :acr, state_variables = :rectangular)
add_modules!(ctx, ACRPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

IVR current-voltage formulation:

ctx = power_context(power; T = 1)
init_power!(ctx; formulation = :ivr, state_variables = :ivr)
add_modules!(ctx, IVRPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

Convex relaxations use the same assembly pattern. For example, W-space SOCP:

ctx = power_context(power; T = 1)
init_power!(ctx; formulation = :w_soc, state_variables = :w_space)
add_modules!(ctx, WSOCPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

8. Inspect Available Components#

Use the registry when you are not sure which component to call:

registered_optimization_modules()
find_optimization_modules(:power_flow)
find_optimization_modules(:storage)
find_optimization_modules(:hvdc)
module_coverage_report()

The registry reports component names, formulations, problem classes, required networks, required variables, and implementation status. It is the quickest bridge from “I want to add this kind of device” to “which PetraFlow component should I call?”
