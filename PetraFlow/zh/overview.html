项目总览#

PetraFlow 是一个面向电力系统稳态分析、最优潮流、模块化优化建模和数值优化算法研究的 Julia 平台。它不是单纯把某个求解器包装成一个函数，而是围绕“电力网络数据如何进入模型”“网络方程如何表达”“设备和约束如何组合”“求解后如何查询结果”这几个问题组织代码。

从用户角度看，PetraFlow 提供两种互补的使用方式。第一种是直接入口，例如 runpf、solve_pf、runopf，适合快速读取算例、运行潮流或最优潮流并查看结果。第二种是模块化建模入口，例如 power_context、init_power!、add_modules!、solve_modular_model!，适合研究不同 ACOPF formulation、凸松弛、HVDC、FACTS、储能、机组组合和安全约束的组合效果。

PetraFlow 解决什么问题#

PetraFlow 当前围绕三条主线组织。三条主线共享同一套网络数据和结果查询接口，但面向的用户任务不同。

主线

目标

典型入口

适合用户

潮流与稳态分析

快速读取算例，计算潮流、导纳矩阵、灵敏度、安全裕度和连续潮流

load_network, runpf, make_ybus, make_ptdf, static_security_analysis

希望像 MATPOWER 一样快速分析系统状态的用户

最优潮流求解平台

运行标准 ACOPF、DCOPF-QP、benchmark，并服务结构化 NLP/QP 求解器研究

需要对照 benchmark、分析求解器表现的用户

模块化建模

需要自定义优化模型或比较 formulation 的用户

直接分析与模块化建模#

如果目标是“我有一个 MATPOWER/PGLIB 算例，希望尽快算出潮流或 OPF 结果”，通常从直接入口开始：

net = load_network("case14")
pf = runpf(net; method = :NR, verbose = false)

如果目标是“我希望同一个网络在极坐标、直角坐标、IVR 或 W-space 松弛中比较结果”，则进入模块化建模：

power = load_power_network("case118")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)
add_modules!(ctx, ACPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())
result = solve_modular_model!(ctx; solver = :ipopt)

这两条路径不是互斥的。直接分析适合快速验证数据、初值和基准结果；模块化建模适合逐步加入设备、限值、成本和跨系统耦合。实际研究中常见的做法是先用 runpf / runopf 确认算例和求解器状态，再用模块化接口构造可解释、可替换、可扩展的优化模型。

模块化建模的核心思想#

PetraFlow 的模块化建模把一个优化问题拆成四类对象：

对象

作用

用户通常需要关心什么

网络 bundle

保存电力、通信、气、热等网络数据

使用 load_power_network 或 build_power_network 准备数据

状态变量空间

决定电压、相角、电流、W-space 或 DC 侧变量如何初始化

在 init_power! 中选择 formulation 和 state_variables

组件

向 JuMP 模型写入变量、约束和目标

用 add_modules! 选择潮流、设备、限值、成本、储能、HVDC 等组件

求解与结果

调用求解器并恢复语义化结果

用 solve_modular_model! 求解，用 bus_solution 等函数查询

这种组织方式让用户不必每次手写完整 OPF 模型。组件自己声明需要哪些变量、提供哪些变量、属于 LP/QP/NLP/SOCP/MILP/MINLP 中哪一类问题；上下文负责把这些组件连接到同一个 JuMP 模型中。

数据、求解器和结果#

PetraFlow 的数据层兼容 MATPOWER/PGLIB 风格字段，同时提供 typed Network 和 builder API。用户既可以直接读取文件，也可以在 notebook 中逐项添加母线、发电机、支路、变压器、shunt、储能和成本。

求解器层通过 JuMP/MOI 接入外部求解器。NLP 类 ACOPF 通常使用 Ipopt；LP/QP/SOCP/MISOCP 可使用 Gurobi 或 HiGHS；PetraFlow 内部的 NLP/QP 求解器用于结构化算法研究。模块化接口尽量把求解器选择写成统一参数，便于替换和对照。

结果层强调“语义化查询”。求解后用户不需要直接在 JuMP 变量容器里翻索引，而是可以使用：

solution_status(result)
bus_solution(result; time = 1)
generator_solution(result; time = 1)
branch_solution(result; time = 1)
solution_value(result, :power, :pg; index = 1, time = 1)

推荐阅读路径#

第一次阅读 PetraFlow 文档时，建议按下面顺序进入：

  1. 安装与环境：准备 Julia 环境、确认依赖和求解器。

  2. 快速开始：运行一个潮流、构造一个小网络、求解一个模块化 ACOPF。

  3. 项目架构：理解数据层、模块层、求解器层和结果层的边界。

  4. 模块化优化：学习 power_context、init_power!、add_modules! 的完整流程。

  5. 组件与 formulation：查找不同潮流和设备 formulation 的变量、约束和适用求解器。
