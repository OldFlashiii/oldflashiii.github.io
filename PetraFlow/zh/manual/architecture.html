PetraFlow 架构总览#

PetraFlow 的文档和代码架构借鉴 PowerModels.jl 的组织思想：把“求解什么问题”和“用什么网络 formulation 表达问题”拆开，再用可复用组件把设备、限值、成本、安全约束和多能耦合接入模型。这样做的目的不是把接口变复杂，而是让用户可以在清晰的层级中逐步建模。

三条用户路径#

路径 A：像 MATPOWER 一样直接分析#

如果目标是快速运行潮流或 OPF，可以直接使用高层入口：

pf = runpf("case118"; method = :NR, verbose = false)
opf = runopf("case118"; formulation = :acp, solver = :ipopt)

这条路径适合教学、benchmark、稳态诊断和快速对照。

路径 B：像搭积木一样模块化建模#

如果目标是比较不同 ACOPF formulation，或组合储能、HVDC、FACTS、机组组合、安全约束等组件，使用模块化入口：

power = load_power_network("case118")
ctx = power_context(power; T = 1)

init_power!(ctx; formulation = :acp, state_variables = :polar)
add_modules!(ctx, ACPPowerFlow(), GeneratorLimits(), BranchLimits(), GenerationCost())

result = solve_modular_model!(ctx; solver = :ipopt)

这条路径强调用户友好：用户只选择 formulation、组件和求解器，不必每次手写完整数学模型。

路径 C：研究结构化求解器#

如果目标是研究 KKT 路由、内点法、QP presolve、缩放、恢复阶段或加速策略，使用 PetraFlow 的 solver core 和 benchmark harness。该层可以接收 OPF 应用层或模块化建模层生成的问题，也可以直接接收通用 NLP/QP 问题。

每个组件遵守同一个协议：

函数

作用

component_name(component)

组件名称，用于注册表和报告

formulation_short_name(component)

数学 formulation 名称，例如 PF-AC-Polar

formulation_problem_class(component)

LP、QP、NLP、SOCP、MILP、MISOCP、MINLP 等

required_networks(component)

需要的网络，例如 :power, :cyber, :gas, :heat

required_variables(component)

状态变量和 formulation 的关系#

init_power! 负责初始化网络状态变量。常用组合如下：

formulation

state_variables

变量空间

适用问题

:acp

:polar

Vm, Va

标准 ACOPF、AC 潮流

:acr

:rectangular

Vr, Vi

直角坐标 ACOPF

:ivr

:ivr

Vr, Vi, branch current

IVR 电流-电压模型

:dcp

:dc

Va, Pf

DCOPF、线性潮流

:ptdf

:ptdf

net injection, branch flow

PTDF OPF

:w_soc

:w_space

Wii, Wijr, Wiji

W-space SOCP 松弛

:bf_soc

:voltage_square

v, l, P, Q

Branch-flow SOCP

含 HVDC 或 AC/DC 混联网络时，应显式设置：

init_power!(
    ctx;
    formulation = :acp,
    state_variables = :polar,
    network_type = :acdc,
    dc_state_variables = :voltage_square,
)

默认 network_type = :ac，因此纯交流算例不会创建 DC 变量。
